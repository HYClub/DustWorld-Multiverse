# 微尘世界：多元宇宙 项目检查方案

## 总则

每条操作流必须完成 **全部 6 层检查** 才能进入下一条流。不允许跨流打点。

### 6 层检查矩阵

```
L1 [用户操作]    用户在浏览器里做什么（点击、输入、滚动）
L2 [DOM/事件]    页面 HTML 绑定、事件监听、Shadow DOM
L3 [页面逻辑]    js/pages/* 层：数据组装、调用编排
L4 [数据管理]    js/api/data.js 层：缓存、校验、编排
L5 [API 封装]    js/api/github.js 层：GitHub Content API 交互
L6 [持久化]      state.json / config.json / history.json 字段定义
```

## 第 1 步：画架构地图

不审查细节，先画出所有模块的输入/输出关系。

```
[文件]                    [输入]                [输出]
pages/home.js             world meta[]          world-card DOM
pages/world.js            world state           10个面板渲染
pages/create.js           表单输入              state.json + config.json
pages/user.js             auth user             world-card DOM
api/data.js               api 原始数据          缓存 + 映射后的对象
api/github.js             API endpoint          JSON response
engine/world.js           config + state        演化后的 state
workers/evolve-cron/...   state.json            更新后的 state.json
```

**产出物**: 一张模块依赖图，标注每个模块读/写哪些文件。

## 第 2 步：逐条操作流追踪

每条流固定格式：

### 流模板

```
## 流名称

### L1 用户操作
[用户做了什么]

### L2 DOM/事件
[哪个 HTML 元素，绑定什么事件，事件处理函数在哪]

### L3 页面逻辑
[函数名 → 函数名 → 函数名，参数列表逐字段列出]

### L4 数据管理
[data.js 函数名 → 参数 → 返回值字段列表]

### L5 API 封装
[github.js 函数名 → HTTP 方法 → path → body 字段列表]

### L6 持久化
[写入哪个文件，JSON 结构快照]

### 逐字段一致性检查
[列出一个字段在各层的名字，标记不匹配]

### 错误路径检查
[每个 .then 是否都有 .catch，超时、404、409、422 各怎么处理]

### 边界情况
[空数据、并发、未登录、已删除、特大数据]
```

### 需要检查的 8 条流

1. **首页加载** — 用户打开网站看到世界列表
2. **创建世界** — 用户填 5 步表单提交
3. **世界详情** — 用户点击卡片看到完整状态
4. **点赞** — 用户点击 ♥
5. **干预** — 用户选类型 → 选聚落 → 确认
6. **登录** — 用户点击登录 → OAuth 回调
7. **用户页面** — 用户查看我的世界/点赞/干预历史
8. **删除世界** — 用户确认毁灭

## 第 3 步：通用数据审计

跨流的数据结构，逐字段从创建到读取到更新到删除，列出所有出现过的文件。

### `state.json` 字段生命线

```
字段名           创建(create.js)    读取(data.js)       更新(engine.js)     读取(worker.js)
─────────────────────────────────────────────────────────────────────────────
world_id         [值]              [字段名]            [字段名]            [字段名]
name             [值]              [字段名]            [字段名]            [字段名]
era              [值]              [字段名]            [字段名]            [字段名]
...
```

每个字段标记：
- ✅ 4 层名字一致
- ⚠️ 名字不一致（注明各层名字）
- ❌ 某层缺失

## 第 4 步：错误处理审计

对每个 `request()` / `.then()` / `.catch()` 调用点，记录：

```
文件:行    Promise 链        有无 catch     catch 做什么    可能遗漏的错误
────────────────────────────────────────────────────────────────
x.js:10   getWorldList()    ✅          返回 []          无
x.js:20   getWorldState()   ✅          返回 null        429 Rate Limit?
```

---

## 第 5 步：边界情况检查

每个流列出边界情况，逐条验证：

### 示例：创建世界

| 边界 | 预期 | 实际 |
|---|---|---|
| world_id 碰撞 | 409 提示重试 | ❌ catch 显示"创建失败"但不解释原因 |
| 未登录提交 | 存 pending → 跳登录 | ✅ |
| config.json 写入失败 | state.json 已写 → 半残世界 | ❌ 无回滚 |

---

## 第 6 步：设计合理性评审

每流回答：

1. 这个设计在当前架构（纯 GitHub Pages + GitHub API）下是否合理？
2. 是否有更简单的实现方案？
3. 这个功能的复杂度/价值比是否合适？
4. 如果现在重写，我会怎么设计？

---

## 执行顺序

```
第 1 步 → 架构地图（全局视角，一次完成）
    ↓
第 2 步 → 逐条流检查（逐流完成 6 层，一次一条流）
    ↓ 每完成一条流
第 3 步 → 字段生命线（跨流验证，增量更新）
    ↓ 流全部完成后
第 4 步 → 错误处理审计
第 5 步 → 边界情况清单
第 6 步 → 设计评审
```

## 核心约束

1. **一次只走一条流**。流 A 未完成全部 6 层前，不看流 B 的任何代码
2. **逐字段写下来**。不允许"感觉一致"，要求把字段名写在纸上对比
3. **改完立刻验证**。每条流修复后，检查：a) 语法无错  b) 相邻流没被破坏
4. **记录"为什么当初这么写"**。如果发现奇怪的设计，先假设作者有合理理由，再质疑
